# Design

## Source of truth

- Status: Draft
- Last refreshed: 2026-05-21
- Primary product surfaces: browser login page, browser R2/WebDAV directory manager, WebDAV protocol endpoints.
- Evidence reviewed: `src/index.ts` inline HTML/CSS for login and directory pages; `README.md` Browser login section; no existing design docs, images, or screenshots were found in the repo.

## Brand

- Personality: secure, calm, utilitarian, storage-console-like.
- Trust signals: clear authentication copy, restrained colors, visible success/error messages, predictable destructive actions.
- Avoid: playful clutter, hidden destructive actions, heavy external assets, styles that interfere with WebDAV protocol behavior.

## Product goals

- Goals: make browser file management pleasant for simple R2 object operations while preserving WebDAV compatibility.
- Non-goals: full file-manager app, drag-and-drop uploads, complex preview system.
- Success signals: login is clear, directory actions are easy to discover, mobile layout is usable, WebDAV clients remain unaffected.

## Personas and jobs

- Primary personas: deployer/admin managing a private R2 bucket; occasional browser user; WebDAV client user.
- User jobs: authenticate, browse folders, create folders, rename objects, delete objects, connect via WebDAV client.
- Key contexts of use: public Worker URL over HTTPS, desktop browser, mobile browser for quick fixes, WebDAV clients.

## Information architecture

- Primary navigation: current directory page with parent link.
- Core routes/screens: `/__login`, directory `GET /.../`, browser actions under `/__actions/*`.
- Content hierarchy: page title and current location first, creation action second, object list and per-object actions third.

## Design principles

- Principle 1: browser UI should feel like a small secure storage console.
- Principle 2: protocol compatibility must not be compromised by visual changes.
- Tradeoffs: inline CSS/JS is acceptable because this Worker has no frontend build pipeline; keep CSP explicit.

## Visual language

- Color: slate surfaces, blue/teal primary gradient, red destructive action, green success, red error.
- Typography: system UI stack with Chinese font fallbacks.
- Spacing/layout rhythm: 8px base rhythm, generous cards, compact row actions.
- Shape/radius/elevation: rounded cards, subtle shadows, hairline borders.
- Motion: small hover/focus transitions only.
- Imagery/iconography: CSS/emoji-like text icons only; no external assets.

## Components

- Existing components to reuse: inline login form, toolbar form, directory entry row, message banners.
- New/changed components: polished shell/card layout, breadcrumb/location pill, entry icon, compact action buttons.
- Variants and states: success, error, danger, secondary, disabled, hover, focus-visible, empty.
- Token/component ownership: `src/index.ts` inline styles are the source until a frontend asset pipeline exists.

## Accessibility

- Target standard: pragmatic WCAG AA for contrast and keyboard navigation.
- Keyboard/focus behavior: visible focus rings; buttons and links remain native elements.
- Contrast/readability: dark text on light surfaces; high-contrast action buttons.
- Screen-reader semantics: messages use status/alert roles; forms keep labels/aria labels.
- Reduced motion and sensory considerations: no required animation.

## Responsive behavior

- Supported breakpoints/devices: desktop and narrow mobile browsers.
- Layout adaptations: rows stack on small screens; actions become full-width.
- Touch/hover differences: hover enhancements are optional; touch targets remain large.

## Interaction states

- Loading: native browser form navigation.
- Empty: show a quiet empty directory message.
- Error: red banner with concise copy.
- Success: green banner with concise copy.
- Disabled: lower opacity and not-allowed cursor for reserved paths.
- Offline/slow network, if applicable: native browser behavior.

## Content voice

- Tone: concise, direct, Chinese UI copy for browser controls.
- Terminology: “文件夹”, “重命名”, “删除”, “退出登录”, “Basic Auth”.
- Microcopy rules: explain security caveats only where they affect user action.

## Implementation constraints

- Framework/styling system: no frontend framework; HTML/CSS/JS generated by Worker TypeScript.
- Design-token constraints: keep styles self-contained; do not add dependencies.
- Performance constraints: no external fonts, images, or large scripts.
- Compatibility constraints: WebDAV methods and auth behavior must remain protocol-compatible.
- Test/screenshot expectations: typecheck, format check, and HTML smoke tests for key UI affordances.
